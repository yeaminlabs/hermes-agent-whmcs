# LLM Provider Management

Hermes Agent supports multiple LLM backends per deployment. This page documents each supported provider, exactly how credentials flow from WHMCS into the running container, and the security model around them.

## Supported Providers

| Provider Key | Friendly Name | `.env` Variable(s) | Notes |
|---|---|---|---|
| `nous_portal` | Nous Portal (recommended) | `NOUS_PORTAL_API_KEY` (only written if a key is supplied) | Default provider; combines Nous-hosted models + built-in tools |
| `openrouter` | OpenRouter | `OPENROUTER_API_KEY` | Aggregator giving access to many third-party models |
| `openai` | OpenAI | `OPENAI_API_KEY` | Direct OpenAI API |
| `anthropic` | Anthropic | `ANTHROPIC_API_KEY` | Direct Anthropic API |
| `custom` | Custom OpenAI-compatible endpoint | `OPENAI_API_KEY` + `OPENAI_API_BASE` | Any self-hosted or third-party OpenAI-compatible server (e.g. vLLM, LiteLLM, Open WebUI-fronted models) |
| *(bedrock, addon-only)* | Amazon Bedrock (Nvidia Models) | `AWS_BEARER_TOKEN_BEDROCK` (always written, hardcoded) | The addon's default Configurable Option; see caveat below |

### Provider ↔ `.env` Mapping (from `hermesagent_CreateAccount`)

```php
if ($llmProvider === 'openrouter') {
    $envLines[] = "OPENROUTER_API_KEY=" . $providerApiKey;
} elseif ($llmProvider === 'openai') {
    $envLines[] = "OPENAI_API_KEY=" . $providerApiKey;
} elseif ($llmProvider === 'anthropic') {
    $envLines[] = "ANTHROPIC_API_KEY=" . $providerApiKey;
} elseif ($llmProvider === 'custom') {
    $envLines[] = "OPENAI_API_KEY=" . $providerApiKey;
    $envLines[] = "OPENAI_API_BASE=" . $customEndpointUrl;
} elseif ($llmProvider === 'nous_portal' && !empty($providerApiKey)) {
    $envLines[] = "NOUS_PORTAL_API_KEY=" . $providerApiKey;
}
```

Every deployment's `.env` **always** includes a hardcoded `AWS_BEARER_TOKEN_BEDROCK` value regardless of the selected provider — this powers the always-on "Nvidia Nemotron via AWS Bedrock" free-tier default referenced throughout the client area (see [Client Area § AWS Bedrock Trial Quota](client-area.md)). This bearer token is identical across **all** deployments (it's a literal string in the module source, not generated per-service) — see [Security Considerations](#security-considerations) below.

### Bedrock / Nemotron Default Path

The addon's One-Click Product Setup only exposes one **LLM Provider** configurable-option choice out of the box: `bedrock|Amazon Bedrock (Nvidia Models)`. This value doesn't match any of the `if`/`elseif` branches in `hermesagent_CreateAccount()` above (`openrouter`/`openai`/`anthropic`/`custom`/`nous_portal`), so when a customer selects it, **no provider-specific env var is written** beyond the always-present `AWS_BEARER_TOKEN_BEDROCK`. This is intentional: the shared Bedrock token is the credential, and `config.yaml`'s `model:` field is expected to reference `nvidia.nemotron-nano-3-30b` for it to work. Clients switching away from this default (e.g. to OpenRouter) need to both supply an API key **and** update the model string via [Manage LLM Providers](client-area.md#manage-llm-providers-page-manage_llmtpl).

## Config Storage on the VPS

Each service gets an isolated directory: `/srv/hermes/<serviceid>/data/`, containing:

### `.env`

```bash
# Generated by WHMCS hermesagent module
HERMES_DASHBOARD=1
HERMES_DASHBOARD_HOST=0.0.0.0
HERMES_DASHBOARD_PORT=9119
HERMES_DASHBOARD_BASIC_AUTH_USERNAME=<dashboard_username>
HERMES_DASHBOARD_BASIC_AUTH_PASSWORD=<dashboard_password>
HERMES_DASHBOARD_BASIC_AUTH_SECRET=<dashboard_secret>
AWS_BEARER_TOKEN_BEDROCK=<hardcoded_shared_token>
# + one provider-specific key block (see table above)
# + API_SERVER_* vars if enabled
# + TELEGRAM_BOT_TOKEN / DISCORD_BOT_TOKEN / SLACK_BOT_TOKEN if configured
```

File permissions are set to `600` immediately after being written (`chmod 600`).

### `config.yaml`

```yaml
model: "<llm_provider>/<model_name>"
dashboard:
  show_token_analytics: true
tool_loop_guardrails:
  warnings_enabled: true
  hard_stop_enabled: true
  hard_stop_after:
    exact_failure: 5
    idempotent_no_progress: 5
terminal:
  backend: docker
```

`model` is a slash-joined string combining the provider key and model name at creation time (e.g. `openrouter/hermes-4-405b`), but can later be freely overwritten with any string via the [Manage LLM Providers](client-area.md#manage-llm-providers-page-manage_llmtpl) page — WHMCS does not validate the format.

## Applying Changes — Container Restart Requirement

**Every** credential or model change requires a full `docker restart hermes-<serviceid>` to take effect, since the container reads `.env` only at process start via `--env-file`. Both `hermesagent_ChangePassword()` and `hermesagent_update_llm()` trigger this restart automatically. There is no hot-reload path — expect a few seconds of downtime (dashboard + API + messaging bots) on every config change.

## Messaging Platforms

| Platform | `.env` Variable | Source |
|---|---|---|
| Telegram | `TELEGRAM_BOT_TOKEN` | BotFather-issued token |
| Discord | `DISCORD_BOT_TOKEN` | Discord Developer Portal bot token |
| Slack | `SLACK_BOT_TOKEN` | Only settable via `CreateAccount`'s Configurable Option (`configoption5`/`configoption6`) — the client-area **Manage LLM Providers** page does not expose a Slack field, so Slack tokens can only be set at initial checkout or via direct admin/SSH edits, not self-serviced afterward |

Messaging is optional; if `messaging_platform` is `None` or the token is empty, no messaging env vars are written.

## OpenAI-Compatible API Server

When **Enable OpenAI-Compatible API** is `yes`:

```bash
API_SERVER_ENABLED=true
API_SERVER_HOST=0.0.0.0
API_SERVER_PORT=8642
API_SERVER_KEY=<api_key>
API_SERVER_CORS_ORIGINS=*
```

`API_SERVER_CORS_ORIGINS=*` allows cross-origin requests from any domain — necessary for the [client-area chat widget](client-area.md#main-dashboard-clientareatpl) to work from arbitrary client websites, but means the API accepts browser-based requests from anywhere as long as the caller has the API key.

## Security Considerations

- **Plaintext secrets in the WHMCS database.** `dashboard_password`, `dashboard_secret`, and `api_key` are stored unencrypted in `mod_hermesagent_instances`. Provider API keys and messaging tokens are **not** persisted in this WHMCS table at all — they live only in the remote `.env` file and in WHMCS's own encrypted Custom Fields / Configurable Options storage (client-supplied values), and are read back live over SSH when needed (e.g. for the Manage LLM page).
- **Shared Bedrock token.** The `AWS_BEARER_TOKEN_BEDROCK` value is a single hardcoded string baked into the module source and written identically into every deployment's `.env`. Anyone with SSH/container access to **any** client's instance, or read access to the module source, obtains a credential valid across **all** deployments on **all** VPS hosts running this module version. Treat this token as sensitive infrastructure secret, not a per-client secret — rotating it requires a module code change and redeploying `.env` to every existing instance (e.g. via **Force Redeploy** per service).
- **Log redaction.** `hermesagent_CreateAccount()` explicitly redacts `providerApiKey`, `messagingToken`, `dashboardPassword`, `dashboardSecret`, and `apiKey` from the command text before calling `logModuleCall()`. Other actions (`ChangePassword`, `update_llm`) pass secrets as the `data2`/`data3` params to `logModuleCall()`, which WHMCS's own admin UI is expected to mask — verify this behavior if running an older WHMCS version, since masking is a WHMCS-core responsibility, not something this module fully controls for every call site.
- **Client-side exposure of the API key.** As noted in [Client Area](client-area.md#website-live-chat-widget), the `api_key` is embedded directly in a public-facing JS widget snippet clients are encouraged to paste onto their own websites — this key should be considered public once a client deploys the widget.
- **`escapeshellarg()` usage.** Values written into `.env` via the update-LLM flow are wrapped in `escapeshellarg()` before being echoed remotely, mitigating shell injection from API keys/tokens containing special characters. The initial `CreateAccount` heredoc-based `.env` write does **not** use `escapeshellarg` (it relies on a quoted heredoc, `cat << 'EOF'`, which is safe against shell expansion since the delimiter is quoted) — this is safe as long as the content never contains a line matching `EOF` alone, which is an accepted (very low) risk for API keys.

## Recommended Practices

- Rotate the shared `AWS_BEARER_TOKEN_BEDROCK` periodically since it's a shared secret across your entire fleet — plan for a bulk redeploy when doing so.
- Advise clients that the **API_SERVER_KEY** is effectively "public" once used in the website widget; if they need a private key for other API integrations (e.g. LobeChat, Open WebUI), they should generate a **separate** provider-side key rather than relying on `API_SERVER_KEY`. (This module has no built-in support for multiple distinct API keys per service — there's exactly one `api_key` per instance.)
- Prefer BYO keys (`openrouter`/`openai`/`anthropic`/`custom`) for clients running significant volume, since the shared Bedrock/Nemotron trial is time- and token-limited (15 days / 1.2B tokens per the client-area UI).
